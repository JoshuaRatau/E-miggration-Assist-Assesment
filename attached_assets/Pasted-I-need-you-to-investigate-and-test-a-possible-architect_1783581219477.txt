I need you to investigate and test a possible architecture / workflow miscommunication in the EMA Leads Funnel application.

Important context:
- The EMA Leads Funnel is a separate application from the main E-Migration Assist platform.
- The funnel has its own frontend, backend, API, and PostgreSQL database.
- The funnel is the sender system: it captures leads, qualifies them, manages them in the admin CRM, and only hands them off onward through the referral / conversion boundary.
- It is NOT supposed to behave as if it is the main EMA application.
- The main EMA application is a separate system and should only become relevant at the correct integration point.

Architecture expectations we are working from:
1. Funnel owns:
   - public lead capture
   - assessment flow
   - OTP verification
   - finalize flow
   - thank-you / reference generation
   - admin CRM lead visibility
   - lead qualification and pipeline movement
   - conversion readiness logic
   - referral preparation
2. Main EMA platform should only be involved at the referral / conversion boundary where that is explicitly intended.
3. We suspect the current test or implementation may be incorrectly trying to “find the company” in the main EMA application database during funnel behavior where it should instead rely on the funnel’s own data.

Your task:
I want you to inspect the codebase and determine whether the funnel is incorrectly depending on the main EMA application database or main-app company lookup too early in the flow.

  1. purely local to the funnel database,

  - which screens/actions trigger the lookup,

  1. expected by design,
