# Build Prompt — EMA Leads Funnel → EMA Referral Tunnel (Sender Side)
> This is self-contained: this project does NOT share code or a database with the
> main EMA operating system. The main EMA receiving side already exists in a
> separate project — do not rebuild it. Your job is the sender side.
## Read first — who you are (name vs. role)
You are the "EMA Leads Funnel" — the SENDER. There are TWO separate systems that
share branding:
- The main EMA operating system (E-Migration Assist) — the RECEIVER, in a
  different project. It owns /api/referrals/ingest, the real /referral-gate, the
  callback sender, REFERRAL_TUNNEL_SECRET, and the referral tables.
- The EMA Leads Funnel (this project) — the SENDER. You capture public leads and
  forward them to the main EMA over HTTP.
Do NOT build any receiving side. You only call those on the main EMA. Every
reference to "EMA" below means the separate main EMA you send leads to.
## 0. Mission
Capture immigration enquiries from the public, obtain applicant consent, match to
a partner firm, and hand the referral to the main EMA. EMA links the firm, runs a
mandatory legal conflict check, and — only once that passes — converts the
referral into a client + case, calling you back at each state change.
Two-system tunnel joined by ONE shared HMAC secret; never share a database.
Interop is ONLY: (1) a signed one-time redirect token (funnel → firm browser →
EMA), (2) a signed S2S applicant push (funnel → EMA), (3) a signed S2S status
callback (EMA → funnel). Everything signed with REFERRAL_TUNNEL_SECRET
(HMAC-SHA256) and FAILS CLOSED when the secret is missing.
## 1. What the main EMA already does (do NOT rebuild)
- Accepts your applicant push; upserts the referral (idempotent by referralId).
- Renders /referral-gate?token=..., consumes the token ONCE, forces a
  non-registered firm through EMA account creation/login.
- Owns firm accounts, subscription/entitlement, the conflict check, client and
  case creation (reuses EMA's existing systems; no parallel billing).
- Access ladder: firm sees a REDACTED preview first; full applicant PII + opening
  the case unlock ONLY after the conflict check passes — a paid subscription does
  NOT bypass the conflict check.
- Calls your callback at every state change with emaFirmId, emaClientId, emaCaseId.
You build the mirror: consent, matching, preview, token issuance, applicant push,
and the callback receiver.
## 2. Environment variables (funnel side)
- REFERRAL_TUNNEL_SECRET — shared HMAC secret, byte-identical to EMA's. Missing ⇒
  fail closed (do not send/redirect).
- EMA_APP_URL — base URL of the main EMA. Redirect firms to
  {EMA_APP_URL}/referral-gate?token=<token>; push to
  {EMA_APP_URL}/api/referrals/ingest.
EMA separately holds FUNNEL_API_URL (your callback base) and FF_PARTNER_REFERRALS
(its flag). You don't set those; you just expose the callback route.
## 3. THE CONTRACT (byte-exact — get this wrong and every signature fails)
All three message types are HMAC-SHA256 signed with REFERRAL_TUNNEL_SECRET.
Signatures are base64url (base64 with + → -, / → _, trailing = stripped) — NOT hex.
### 3.0 Shared primitives (reproduce EMA exactly)
    import crypto from "crypto";
    function b64url(buf) {
      return buf.toString("base64").replace(/\+/g,"-").replace(/\//g,"_").replace(/=+$/,"");
    }
    function hmac(data, secret) {
      return b64url(crypto.createHmac("sha256", secret).update(data).digest());
    }
    // Deterministic JSON: recursively sort keys so both systems produce IDENTICAL
    // bytes. Used for the S2S bodies.
    function stableStringify(value) {
      if (value === null || typeof value !== "object") return JSON.stringify(value);
      if (Array.isArray(value)) return `[${value.map(stableStringify).join(",")}]`;
      const keys = Object.keys(value).sort();
      return `{${keys.map(k => `${JSON.stringify(k)}:${stableStringify(value[k])}`).join(",")}}`;
    }
Two serializations are used ON PURPOSE:
- Redirect token body is signed over base64url(JSON.stringify(payload)) (plain
  JSON.stringify then base64url — see 3.1).
- S2S bodies (ingest push, callback) are signed over stableStringify(body)
  (recursively key-sorted — see 3.2 / 3.3).
### 3.1 Redirect token (funnel → firm browser → EMA)
Token string = "<body>.<sig>" where body = base64url(utf8(JSON.stringify(payload)))
and sig = hmac(body, secret)  ← HMAC over the base64url body STRING.
Payload (NEVER put applicant PII here):
    {
      "referralId": "string",
      "assignmentId": "string?",
      "funnelFirmId": "string?",
      "intendedAction": "accept_referral_open_ema",  // MUST be exactly this literal
      "issuedAt": 0,        // epoch MILLISECONDS
      "expiresAt": 0,       // epoch MILLISECONDS — keep short (e.g. now + 30 min)
      "nonce": "string"     // unique per token, e.g. randomBytes(16).toString("hex")
    }
EMA rejects on: signature mismatch, now > expiresAt, intendedAction not exactly
accept_referral_open_ema, or replay after consumption. Treat as one-time-use on
your side too; honour expiresAt.
    function signReferralToken(payload, secret) {
      const body = b64url(Buffer.from(JSON.stringify(payload), "utf8"));
      return `${body}.${hmac(body, secret)}`;
    }

### 3.2 Applicant push (funnel → EMA, S2S)
POST {EMA_APP_URL}/api/referrals/ingest
Headers: content-type: application/json, x-referral-signature: <sig>
where sig = hmac(stableStringify(body), secret).

      "funnelAssignmentId": "string|null", // NOTE: funnelAssignmentId (NOT assignmentId here)
      "funnelFirmId": "string|null",

### 3.3 Status callback (EMA → funnel, S2S) — YOU expose this
POST {FUNNEL_API_URL}/api/referral-gate/callback
EMA sends x-referral-signature: <sig> = hmac(stableStringify(body), secret).
You MUST verify it and reject on mismatch. Best-effort on EMA's side (5s timeout)
— make your handler fast and idempotent.
    {
      "referralId": "string",
      "assignmentId": "string|null",  // your id echoed back (EMA echoes funnelAssignmentId)
      "status": "ema_account_required | ema_account_linked | converted | failed",
      "emaFirmId": "string|null", "emaClientId": "string|null", "emaCaseId": "string|null",
      "reason": "string|null"         // e.g. "conflict_of_interest" when status = failed
    }
    function verifyBodySignature(rawStableBody, signature, secret) {
      if (!secret || !signature) return false;             // fail closed
      const expected = hmac(rawStableBody, secret);
      const a = Buffer.from(signature), b = Buffer.from(expected);
      return a.length === b.length && crypto.timingSafeEqual(a, b);
    }
    // handler: verifyBodySignature(stableStringify(req.body), req.header("x-referral-signature"), secret)

## 5. Hard rules (non-negotiable)
- Fail closed: missing REFERRAL_TUNNEL_SECRET ⇒ do not sign, push, or redirect.
- No applicant PII in: the redirect token, preview fields, referral emails, or any
  audit/callback log. PII travels ONLY inside the signed applicant object of §3.2.
- Consent before data: never push applicant data (§3.2) without recorded POPIA
  consent to share with the receiving firm.
- Tokens are one-time-use and short-lived; honour expiresAt.
- intendedAction must be exactly accept_referral_open_ema or EMA rejects the token.
## 6. End-to-end acceptance criteria (final gate)
- Funnel sends a consented referral into EMA.
- EMA consumes the token only once.
- Missing/invalid HMAC fails closed (rejected, nothing leaks).
- Non-registered firm is forced through EMA registration.
- Full applicant details unlock ONLY after the conflict check passes.
- Subscription gates still apply to full EMA features.
- Funnel receives the conversion callback with emaFirmId and emaCaseId.
- Funnel admin board shows the referral as converted_to_ema_case.
- No applicant PII in the referral email, redacted preview, token payload, or
  unsigned logs.
