# EMA Leads Funnel

**Internal/operational name for this repo, workspace, and admin back-office.** This app is the lead-capture funnel (the "sender") that feeds the *separate* main **E-Migration Assist** platform. The public consumer-facing brand shown to leads and visitors (capture pages, referral preview, emails, WhatsApp) intentionally stays "E-Migration Assist" — only the internal/operational identity is "EMA Leads Funnel". Go by ROLE, not name: this repo is a funnel, not the main EMA platform.

E-Migration Assist (the public brand) helps users navigate the immigration process via a 5-step assessment, optional document uploads, and WhatsApp integration, paired with an admin CRM for lead management.

## Run & Operate

`pnpm install` / `pnpm dev` / `pnpm build` / `pnpm typecheck` / `pnpm orval` / `pnpm db:push`

**Required env:** `DATABASE_URL`, `RESEND_API_KEY`, `TWILIO_ACCOUNT_SID`, `TWILIO_AUTH_TOKEN`, `TWILIO_PHONE_NUMBER`, `OTP_SECRET`, `REPLIT_OBJECT_STORAGE_URL`.
**Optional:** `ADMIN_EMAIL_TOKEN` (legacy `x-admin-token`), `BOOTSTRAP_ADMIN_EMAIL`/`PASSWORD` (override seeded `demo@admin.local` / `ChangeMe!2026`), `PUBLIC_BASE_URL` (falls back to `https://$REPLIT_DEV_DOMAIN`; required for unsubscribe footer in email campaigns), `WHATSAPP_APP_SECRET`/`WHATSAPP_VERIFY_TOKEN` (preferred webhook signature path; falls back to `TWILIO_AUTH_TOKEN` HMAC), `UNSUBSCRIBE_SECRET` (HMAC; falls back to `SESSION_SECRET`; fails closed in prod if both unset).

## Stack

Express 5 + React, Node 24, TS 5.9, Drizzle, Zod, esbuild, PostgreSQL. pg-boss v12 for background jobs.

## Architecture decisions

Ordered by milestone, oldest → newest. Milestones 1–3 are condensed to objective / major decisions / final implementation / lessons; Milestones 4–5 are kept in full phase detail. Anything not captured here is either obvious from the code or in `git log`.

### Milestone 1 (Funnel Foundation) — public assessment funnel + admin CRM + bulk-comms engine

**Objective:** stand up the public lead-capture assessment funnel, the admin CRM that manages the resulting leads/cases, and the first bulk email/WhatsApp campaign engine.

**Major decisions:**
*   **Two-phase lead submission:** the lead row is committed at the Terms step with `finalize:false` (no email/WA dispatched); the confirmation is sent only when `POST /api/leads/:id/finalize` fires, and is at-most-once (skips if a `lead_engagements` `confirmation` row already exists). `finalize` defaults to `true` for back-compat.
*   **Reference number is post-finalize-only:** generated server-side at insert but revealed only at `/thank-you/:reference` — never inside the assessment flow.
*   **Public assessment UX:** dynamic 7/8 steps (upload gate), session-scoped document listing (`sessionStartedAt` cutoff so returning users never see prior-session docs), inside/outside-SA residence handling, OTP prioritising WhatsApp then falling back to email, idempotent lead→case conversion.
*   **Admin auth (email + password):** `admin_users`/`admin_sessions`/`admin_password_resets`; login mints an opaque server-side session in an httpOnly `ema_admin_session` cookie (7-day TTL, secure in prod); `requireAdminAuth`/`requireAdminToken` check cookie first, fall back to legacy `x-admin-token`. Demo superadmin (`demo@admin.local`) seeded if `admin_users` empty; forgot-password mints a 1-hour single-use hashed token; `/admin/users` superadmin-only with self-protection. New admin fetches use `credentials:"include"`.
*   **Admin shell:** single `admin-layout.tsx` chrome — no left sidebar, `AdminUserMenu` is the sole nav surface covering all modules.
*   **Bidirectional lead pipeline:** statuses move both ways; the single hard invariant is that entering `converted` requires current status `ready_for_case` (the same PATCH triggers `ensureCaseForLead`) — enforced atomically in the PATCH WHERE-clause (closes TOCTOU, 409 on violation) and mirrored in `lib/leadStatus.ts → canAdvanceStatus`. Case statuses stay forward-only. Optimistic UI with server re-validation + rollback.
*   **Campaign engine (Phase 4):** `campaigns`/`campaign_recipients`/`unsubscribes`; a zod-validated single-level AND/OR audience-query compiler over a 12-field whitelist (32-rule cap); 4 merge tokens (`{{first_name}} {{full_name}} {{reference}} {{organization_name}}`); HMAC-signed RFC-8058 one-click unsubscribe; atomic `draft→sending` claim.
*   **Executive Dashboard / CRM (Phase 5):** `prelaunch_leads` extended (`lead_type`, `inquiry_type`, `source`, `assigned_to` uuid soft-ref to `admin_users` with no FK, `last_contacted_at`, `next_follow_up_at`, `tags[]`, 10 professional-lead cols); slim `AdminLeadListItem` + public-safe `PublicLead` serializers avoid over-fetching; `/admin/communications` four-tab hub; reusable `comm_templates` (subject required for email, channel locked after create, archived rows immutable, preview via `campaignRender.ts`).

**Final implementation:** a working public 5-step assessment with optional document upload + WhatsApp/email engagement + public status lookup, an admin CRM (leads + cases, kanban, forward-only case lifecycle), and a functional campaign engine (email + in-window WhatsApp).

**Lessons / durable rules:** every privileged admin mutation writes an append-only `lead_audit` row (actor credential sha256-hashed, raw never stored) — this audit trail is the backbone reused by later notes/timeline features; the `website` honeypot field must never be rendered (API silently rejects it with a synthetic 201); the document allow-list is server-side only; public APIs minimise PII and are rate-limited; the WhatsApp webhook is fail-closed (503 without `WHATSAPP_APP_SECRET`/`TWILIO_AUTH_TOKEN`).

### Milestone 2 (Funnel Intelligence) — scoring, segmentation, richer campaigns, automation scaffold, sender-side referral tunnel

**Objective:** layer commercial intelligence and automation onto the foundation — lead scoring, tier intent, source attribution, richer/queued/scheduled campaigns, a lifecycle-automation scaffold, and the sender-side referral tunnel into the separate EMA platform.

**Major decisions:**
*   **Tier-aware intent (6A.5):** nullable `prelaunch_leads.intended_tier` (11 allow-listed values across B2C/B2B/white-glove motions); enforced at the API layer; the public insert path was not touched.
*   **Event-sourced lead scoring (6B):** append-only `lead_events` stream + a 60s recompute worker replaces the synchronous heuristic. Rubrics live in code (`scoringRubrics.ts`) and the points-per-event are snapshotted into each event row so historical contributions stay immutable. Wired at lead-created / assessment-completed / status-advanced / tier-set. Accepted V1 gap: ~60s lag before a score refreshes.
*   **Funnel trim (6A.1):** lead pipeline reduced 10→9 stages (dropped `awaiting_response`, duplicated by `contacted` + `next_follow_up_at`).
*   **B2B contact intelligence (6A):** representative role/relationship derived from the email local-part + organisation type; hover-card contact card for `professional` leads; slim serializer extended to avoid N+1.
*   **Campaign hardening:** rich-text TipTap composer with a DOMPurify sanitiser run on BOTH write and render (6D-2); a **pg-boss v12** background queue for sends — `POST /send` returns **202** + polling, 2000-recipient cap, atomic per-recipient claim + single-winner finaliser (6D-3A); scheduled send + pause/resume via a shared `dispatchClaimedCampaign` path that re-evaluates the audience query at fire time, with a 30s single-replica scheduler tick (6D-3B); per-channel SLA due-date fields (6D-1).
*   **Source attribution (6E):** a CRM-style attribution panel backed by a bespoke `GET /api/stats/source-attribution` — deliberately outside OpenAPI until the shape stabilises; all inputs allow-list-validated before any `sql.raw`.
*   **Lifecycle automations scaffold (6F-4a):** `lifecycle_rules` + `lifecycle_executions` (the latter's `UNIQUE(rule_id, lead_id, triggered_by)` makes duplicate event delivery a no-op). 3 starter rules seeded **all disabled** via `onConflictDoNothing`; read-only API only — no worker, no mutations, no side effects this phase.
*   **Referral tunnel (SENDER only — never build the receiver here):** captured leads, on explicit POPIA consent, are offered as a **redacted** referral to a matched partner firm that accepts inside the separate main EMA platform. Byte-exact HMAC contract in `lib/referralTunnel.ts` with **two deliberate serializations** (redirect-token body vs. server-to-server key-sorted body). Fail-closed: no `REFERRAL_TUNNEL_SECRET` ⇒ 503. **PII invariant (structural):** the `referrals` table has NO name/email/phone columns — applicant PII travels only inside the signed push body. Consent creation locks the lead row `FOR UPDATE`; the `converted` callback is terminal/idempotent.
*   **Canonical email sender:** all platform email sends FROM `noreply@emigration-assist.com` (`EMAIL_FROM` || spec constant); the domain must be verified in Resend (SPF/DKIM) or sends bounce.
*   **Support Centre mirror:** support-widget submissions are fire-and-forget forwarded to the external Eride Support Hub, with the returned reference persisted back onto the local `support_requests` row.

**Final implementation:** intelligent lead scoring + tier/attribution analytics on the dashboard, a queue-backed schedulable campaign engine with a rich composer, a dormant lifecycle-rule scaffold, and a live, PII-safe referral hand-off to the EMA platform.

**Lessons / durable rules:** keep the PUBLIC brand "E-Migration Assist" while the internal/operational identity is "EMA Leads Funnel" (go by ROLE, not name); pg-boss requires a Reserved VM (autoscale breaks the queue); bespoke analytics/referral routes are intentionally kept out of OpenAPI to avoid codegen churn; campaign sends now return 202 (poll for counters), pause is best-effort (in-flight batch still drains), and scheduled campaigns stay editable until claimed.

### Milestone 3 (Lead Operations) — status nuance, internal notes, ownership, follow-ups, archive/delete

**Objective:** turn captured leads into workable CRM records — status nuance, internal notes, ownership, follow-ups, and lifecycle removal — deliberately REUSING existing tables/columns (no new task table).

**Major decisions:**
*   **Status (11A):** added `needs_more_information` to the 9-stage `lead_status` pipeline. `lead_status` is a text col (no migration), but the value is mirrored across ~9 hardcoded enumeration sites incl. a type-required kanban `Record`; OpenAPI carries no `enum:` so no codegen.
*   **Internal notes (11B):** stored as append-only `lead_audit` rows (`action="lead_note_added"`, text in `after.note`) — reusing the audit mechanism so notes auto-surface in the shared timeline with actor + timestamp. Durability contract: the POST **awaits** the insert and 500s on failure (user-intent data must never be silently lost). Coexists with the pre-existing single-blob `adminNotes` (blob = editable summary; notes = append-only authored log).
*   **Ownership (11C):** reuses the pre-existing `assigned_to` (uuid soft-ref to `admin_users`, NO FK). Roster via `GET /api/admin/assignable-users` (any admin). Active-assignee rule: a *change* to a deactivated user is rejected, but re-PATCHing a lead already owned by a now-inactive user is allowed (compares against the before snapshot). The audit snapshots the assignee NAME so the timeline renders join-free.
*   **Follow-ups (11D):** a per-lead "Next follow-up" reuses `next_follow_up_at` + ONE new `follow_up_note`. The owner is DERIVED from `assignedTo` (never a new field). Invariant: a note may never outlive its due date — if the resolved due-date is null, `follow_up_note` is force-nulled. Audit action is resolved from the before/after due-date transition (scheduled/updated/removed); completion is a dedicated sibling route that also stamps `lastContactedAt`.
*   **Archive / delete:** soft-archive via `prelaunch_leads.archived_at` (`GET /api/leads` excludes archived by default). Delete is atomic — lead-exists + no-linked-case checks run inside the txn with `SELECT … FOR UPDATE` (409 if a case exists), cascades child rows but keeps the append-only `lead_audit`.

**Final implementation:** per-lead status / notes / owner / follow-up editing on both the lead-detail page and the dashboard drawer, an "Assigned To" filter chip, activity-feed summaries for every new verb, and Active/Archived views with archive/restore/delete.

**Lessons / durable rules:** `adminNotes` is omitted from the slim `GET /api/leads` list — inline/drawer note editors must fetch the full lead via `/api/leads/by-id/:id` (key `["admin","lead",id]`); adding a `lead_status` value means touching ~9 hardcoded sites incl. the kanban Record (see `.agents/memory/lead-status-enum-sites.md`).

### Milestone 4 (EMA Platform Integration) — Phase 13C "Activate Client Portal" action + 13B "Prepare Client Portal" action + 13A client-portal preparation (read-only) + 12C workflow auto-attach + 12B convert action + 12A conversion mapping

**13C — "Activate Client Portal" admin action (the forward-only follow-up to 13B; still no client-facing side effects):** an admin-only action transitions a PREPARED case (`portal_status='ready_to_activate'`) to `activated`. **STILL no client-facing side effects:** issues NO credentials/login, sends NO email/WhatsApp, exposes NOTHING publicly, does NOT touch the public status/reference lookup or expose documents/workflow info — it flips ONE status column so a FUTURE phase can wire up real client access. **Lib (`cases.ts → activateCasePortal(caseId)`):** mirrors `prepareCasePortal` — a single transaction with `SELECT … FOR UPDATE` on the case row. **Order-sensitive gate:** `activated` is checked FIRST (idempotent success, never downgraded) → then workflow_status MUST be `assigned` (else `blocked_review`) → then portal_status MUST be `ready_to_activate` (else `blocked_not_ready`); both blocks leave state UNTOUCHED. **Only allowed transition:** `ready_to_activate` → `activated` (`activated`, `changed=true`). Returns `{outcome, changed, portalStatus, previousPortalStatus, workflowStatus}`. **IDEMPOTENT & race-safe:** the read-decide-write is atomic under `FOR UPDATE` and `changed` is true ONLY for the single call that transitions the row, so concurrent callers audit `portal_activated` at most once and re-runs are silent success. **Route (`POST /api/admin/leads/:id/activate-portal`, `requireAdminToken`, NOT in OpenAPI — sibling of /prepare-portal, /convert, /notes):** 404 if lead missing → 422 if no linked case (`not converted`) → run `activateCasePortal` → on `blocked_review`/`blocked_not_ready` audit `portal_activation_blocked` (reason `workflow_not_assigned` / `portal_not_ready`) + return **422** (portal untouched) → on `changed` audit `portal_activated` ONCE (before=`previousPortalStatus`) → `already_activated` returns 200 with NO audit → try/catch audits `portal_activation_failed` + 500. Every response returns `serializeLead` with the case's workflow+portal state so the client cache stays consistent. **Reuses the existing audit verbs** — `portal_activated` now FIRED for the first time (was reserved since 13A). **Frontend (`admin-lead-detail.tsx`):** the Client Portal card's action area now expresses the full flow — Prepare Portal (when not prepared) → **Activate Portal** (replaces the Prepare button once `ready_to_activate`) → Activated badge (terminal, button hidden). `handleActivatePortal` mirrors `handlePreparePortal`'s fetch/token pattern: 422 → toast + cache update, 200 → cache update + `events` invalidation. `lead-activity-feed.tsx` already summarises `portal_activated` (13A). **Unchanged:** questionnaires/submission/funnel routing/workflow-assignment logic/dashboard/public routes/attribution; no schema change, no OpenAPI, no `db:push`.

**13B — "Prepare Client Portal" admin action (the first WRITE to `portal_status`, still no client-facing side effects):** an admin-only action marks a converted, workflow-ASSIGNED case as `ready_to_activate` for a FUTURE activation. **STILL preparation-only:** creates NO client credentials/login, sends NO email/WhatsApp, exposes NOTHING publicly, does NOT touch the public status/reference lookup, and adds NO "Activate" button — it flips ONE status column. **Lib (`cases.ts → prepareCasePortal(caseId)`):** a single transaction with `SELECT … FOR UPDATE` on the case row. **Gate (authoritative, server-side):** workflow_status MUST be `assigned` — else outcome `blocked_review`, `portal_status` left UNTOUCHED (a human resolves the workflow first; we never prepare an undecided case). **Forward-only state machine:** `not_prepared`/`manual_review_required` → `ready_to_activate` (`prepared`, `changed=true`); `ready_to_activate` → no-op (`already_ready`); `activated` → no-op **terminal, never downgraded** (`already_activated`). **IDEMPOTENT & race-safe:** the read-decide-write is atomic under `FOR UPDATE` and `changed` is true ONLY for the single call that actually transitions the row, so concurrent callers audit at most once and re-runs are silent. Result carries `previousPortalStatus` so the audit records the TRUE pre-transition state (not a hardcoded `not_prepared` — a `manual_review_required`→`ready_to_activate` transition audits honestly). **Route (`POST /api/admin/leads/:id/prepare-portal`, `requireAdminToken`, NOT in OpenAPI — sibling of /convert, /notes, /archive):** 404 if lead missing → 422 if no linked case (`not converted`) → run `prepareCasePortal` → on `blocked_review` audit `portal_activation_blocked` + return **422** (portal untouched) → on `changed` audit `portal_prepared` ONCE → try/catch audits `portal_activation_failed` + 500. Every response returns `serializeLead` with the case's workflow+portal state so the client cache stays consistent. **Uses the Phase 13A audit verbs** (`portal_prepared`/`portal_activation_blocked`/`portal_activation_failed`) — now FIRED for the first time; `portal_activated` remains reserved for the future real activation phase. **Derivation change (Phase 13A revised):** `deriveClientPortalStatus` no longer infers `ready_to_activate` from an `assigned` workflow — an assigned-but-unprepared case now derives to `not_prepared` (eligible, but the admin hasn't acted). Readiness is an EXPLICIT persisted state stamped by this action; only a persisted `ready_to_activate`/`activated` shows as such. This keeps the indicator honest and lets the UI enable the Prepare button for exactly the eligible-but-unprepared cases. **Frontend (`admin-lead-detail.tsx`):** the Phase 13A "Client Portal" card gains a "Prepare Portal" button — disabled unless `caseWorkflowStatus==='assigned'` && `casePortalStatus!=='ready_to_activate'`, hidden once `activated`, with a hint ("Assign a workflow before preparing" when the workflow isn't assigned). `handlePreparePortal` mirrors `handleConvert`'s fetch/token pattern: 422 → toast + cache update, 200 → cache update + `events` invalidation. `lead-activity-feed.tsx` already summarises the portal verbs (13A). **Unchanged:** questionnaires/submission/funnel routing/workflow-assignment logic/dashboard; no schema change, no OpenAPI, no `db:push`.

**13A — client-portal readiness indicator (PREPARATION ONLY, read-only, zero side effects):** converted EMA cases now clearly show whether they are ready for a FUTURE client-portal activation — WITHOUT creating any login/credentials/account, sending any email/WhatsApp, duplicating accounts, or exposing anything to clients. **Existing access model (analysed first):** there is NO client login — public case access is an unauthenticated reference-number lookup (`/status` page → `GET /api/public/status/:referenceNumber`); the persisted `portal_status` is passed at every case-aware call site — the by-id detail LEFT JOIN, `POST /convert` (both the success return AND the already-converted short-circuit — the latter passes the existing case's workflow+portal state so response parity holds and the frontend cache write isn't clobbered), and `PATCH …/:id` status→converted (both created + already-converted branches; **PATCH side-effect discipline preserved** — portal state is only READ from `ensureCaseForLead(...).row.portalStatus`, unrelated edits never mutate portal/workflow state). **`casePortalStatus` IS in OpenAPI** (added to Lead schema, codegen run). **Frontend (`admin-lead-detail.tsx`):** a read-only "Client Portal" Card rendered ONLY when `lead.caseId` — shows case reference (`referenceNumber ?? caseId`), a portal-status Badge (4 states), and a workflow assigned/review Badge; no activate button, no credentials, no client-facing surface. `lead-activity-feed.tsx` summarises the 4 (dormant) portal verbs for future consistency. **Unchanged:** questionnaires/submission/funnel routing/dashboard layout/workflow assignment logic; no client account, no email/WhatsApp, no activation.

### Milestone 4 (EMA Platform Integration) — Phase 12C workflow auto-attach + 12B convert action + 12A conversion mapping

**12C — auto-attach the EMA workflow to a freshly converted case (registry-driven, honest, idempotent):** after a lead becomes a case, the correct workflow is attached automatically from the mapper's `workflowCandidate` — no hard-coded workflow IDs, no guessing. **Schema:** two nullable/defaulted cols on `lead_cases` (`lib/db/src/schema/leadCases.ts`, `pnpm --filter @workspace/db run push`) — `workflow_key text` (stable registry key) + `workflow_status text NOT NULL default 'unassigned'` (∈ `unassigned|assigned|review_required`). **Registry (single source of truth):** `leadToApplication.ts` gains a `WorkflowDefinition` interface + `WORKFLOW_DEFINITIONS` (3 stable keys: `visa_application`, `overstay_appeal`, `travel_entry_assistance`) + exported `resolveWorkflow(key)`; the pre-existing `WORKFLOW_BY_MATTER`/`WORKFLOW_BY_ROUTE` maps now REFERENCE the registry (no divergent duplicate list). **Assignment lib:** `cases.ts → assignWorkflowForCase(caseId, candidate)` resolves the candidate key against the registry → recognised ⇒ `workflow_key`+`status='assigned'`; unknown/null ⇒ `status='review_required'` (**never guesses**). **IDEMPOTENT by construction:** the UPDATE is guarded on `workflow_status='unassigned'` so only the FIRST call for a case transitions it; `changed` is sourced from the atomic `UPDATE … RETURNING` (not a pre-check SELECT) so at most one of N racing callers audits; the no-op path reads back persisted state via `readWorkflowState`. **Wired into BOTH conversion paths** so they behave identically: `POST /api/admin/leads/:id/convert` (runs after `ensureCaseForLead`; the already-converted duplicate guard short-circuits BEFORE this, so it stays side-effect-free) and `PATCH /api/admin/leads/:id` status→converted (dashboard lead-drawer). **PATCH side-effect discipline:** workflow assignment runs ONLY when `ensureCaseForLead` reports `created===true` — an unrelated edit (notes/follow-up/assignment) on an already-converted lead must NOT re-enter assignment and risk transitioning a legacy `unassigned` case; that branch instead surfaces the PERSISTED workflow state from the existing case row with zero writes. **Gate coexistence preserved (NOT unified):** PATCH keeps its 12B `ready_for_case` predecessor lock (it can therefore legitimately reach `review_required`); only `/convert` uses the mapper's `canConvert` gate — deliberately left divergent per the 12B "Coexistence, NOT replacement" contract. **2 new AuditAction verbs:** `case_workflow_assigned`/`case_workflow_review_required` (audited only when `changed`, carrying `{workflowStatus, workflowKey, workflowLabel, reason}` before+after). **Read path:** `serializeLead` in BOTH `routes/leads.ts` + `routes/adminLeads.ts` gains a 3rd param exposing `caseWorkflowKey`/`caseWorkflowStatus` (default null); ### Milestone 4 — Phase 12B convert action + 12A conversion mapping

**12B — "Convert to EMA Application" admin action (built on the 12A mapper):** the authorised-staff action that actually converts a qualified lead. **New sibling route `POST /api/admin/leads/:id/convert`** (`adminLeads.ts`, `requireAdminToken`, NOT in OpenAPI — matches /notes, /archive, /follow-up/complete). **Flow:** load full lead (404 if missing) → duplicate guard (a lead already linked to a `lead_cases` row short-circuits to `{alreadyConverted:true, converted:false, case}` with NO side effect / re-audit) → run `buildConversionPreview(lead)` → **if `readiness.canConvert===false`**: convert NOTHING, audit `lead_conversion_blocked`, return **422** with the full `preview` (sections + readiness + workflowCandidate) → **else**: audit `lead_conversion_started`, atomic `UPDATE … SET lead_status='converted' WHERE id AND lead_status!='converted'` (guards concurrent double-run; touches ONLY `lead_status`+`updated_at` so assignment/notes/audit are preserved), then create the application via the **EXISTING** integration point `ensureCaseForLead(id, referenceNumber)` (idempotent INSERT…ON CONFLICT(lead_id) — the UNIQUE(lead_id) IS the duplicate-conversion prevention), audit `lead_converted` **only when `result.created===true`** (carries `{caseId, applicationId:caseId, workflowCandidate}` — leadId+caseId+workflow+actor+timestamp all on the row). Every leaf is nullable. **HONESTY RULE (never guess):** a field with no source maps to `null` and is reported `missing` — deliberately DIVERGES from `buildApplicantPushBody`'s `splitName` which invents an "Applicant" placeholder; the mapper's `splitNameHonest` returns nulls so the gap surfaces. `dateOfBirth`/`passportNumber`/`destination` are ALWAYS null (funnel structurally can't collect them) → bucketed as `manual`, never guessed. **Missing-data detection:** a declarative `FIELD_CATALOGUE` (single list; drives both section rendering AND readiness) tags each field `required` / `optional` / `manual`; `buildConversionPreview` returns `sections[]` (each with `complete`) + `readiness` = `{canConvert, requiredAvailable, requiredMissing, manualCompletion, optionalAvailable, optionalMissing}`. **Required set:** applicant first/last name, `contact.reachable` (**email OR phone** — each channel normalised INDEPENDENTLY before the OR, so an empty-string email can't shadow a valid phone), `assessment.countryOfResidence`, `route.matterType`, `workflowCandidate.key`. **Workflow candidate:** `deriveWorkflowCandidate` maps `inquiryType`→workflow, falls back to funnel `route`, else returns `{key:null, reason}` (operator picks manually) — never a silent guess. **Audit-CAPABLE, write-FREE:** `buildAuditDescriptor` returns a ready `{action:"lead_conversion_previewed", leadId, after}` payload a later phase can hand to `writeAudit`; the verb is a plain string, deliberately NOT added to the `AuditAction` union, and nothing calls the audit system in this phase.

### Milestone 5 (Client Communication & Onboarding) — Phase 14B "Send Portal Activation Email" action (EMAIL ONLY) + 14A notification PREPARATION

**14B — the first real client-facing SEND, built entirely on the 14A pure preview (EMAIL ONLY — WhatsApp deliberately untouched):** an admin action actually emails a client their portal activation. **Strict scope:** email only — sends NO WhatsApp, creates NO WhatsApp templates, does NOT touch Twilio, creates NO client login/credentials, and does NOT change public-portal/questionnaires/lead-submission/routing/dashboard/workflow logic. **Schema:** ONE nullable col on `lead_cases` (`lib/db/src/schema/leadCases.ts`, `pnpm --filter @workspace/db run push`) — `activation_email_sent_at timestamptz` (null = never sent). It is the SINGLE source of truth for BOTH the exposed sent-timestamp AND idempotency (no new table). **Authoritative gate = the 14A preview** (`buildNotificationPreview`): send is allowed ONLY when `readiness==='ready'` (which itself requires portal_status `activated` + client name + case reference + a reachable channel) AND the EMAIL channel specifically is available AND a subject+body composed — a WhatsApp-only case CANNOT satisfy this phase. **Idempotency (default: BLOCK duplicates, no resend flag):** the send is CLAIMED atomically via `UPDATE lead_cases SET activation_email_sent_at = now() WHERE id = $1 AND activation_email_sent_at IS NULL RETURNING …` — only the first caller gets a row back, so concurrent double-sends are impossible; a second attempt (or a lost race) returns 409. **Rollback-on-failure:** the send fires AFTER the claim; if the provider rejects it (`sendCustomEmail` returns `{ok:false}` — it never throws) the claim is rolled back to null so the admin can retry, and NOTHING is left marked as sent. **Route (`POST /api/admin/leads/:id/send-activation-email`, `requireAdminToken`, NOT in OpenAPI — sibling of /activate-portal, /notification-preview, /convert):** 404 missing lead → 422 no case (`not converted`) → 409 if already sent → build preview → if not allowed audit `email_activation_blocked` (reason `not_ready`) + **422** (+preview, nothing sent) → atomic claim (lost race ⇒ audit `email_activation_blocked` reason `already_sent` + 409) → `sendCustomEmail({to: preview.summary.email.destination, subject: preview.email.subject, text: preview.email.body})` → on `{ok:false}` roll back + audit `email_activation_failed` + **502** → on success audit `email_activation_sent` + 200 `{sent, activationEmailSentAt, lead}`. try/catch rolls back + audits `email_activation_failed` (`send_exception`) + 500. **3 new AuditAction verbs FIRED (`email_activation_sent`/`email_activation_blocked`/`email_activation_failed`)** — ADDED to the `AuditAction` union (contrast 14A's `notification_prepared`/`notification_blocked` which stay OUT of the union because nothing fires them). **PII discipline in audit rows:** the persisted `after` carries only SAFE values — reason CODES (`not_ready`/`already_sent`/`forbidden_phrase`/`provider_error`/`send_exception`), the fixed subject constant, `sentAt`, and static readiness/blocked/missing labels; raw provider/exception strings (which can contain the recipient address) go to `req.log` ONLY, never to the audit JSON. **Read path:** `GET …/notification-preview` now also returns `activationEmailSentAt` (route-level projection, NOT in the pure lib — the lib stays pure) so the UI can render the sent state. **Frontend (`admin-lead-detail.tsx`):** the 14A Notification Readiness card gains ONE "Send Activation Email" button with all four states — **Email sent** (terminal badge + timestamp, button gone), **Ready to send** (button, only when `readiness==='ready'` && email available), **Not ready** / **Blocked** (hint text: activate the portal / resolve missing info / no valid email). `handleSendActivationEmail` mirrors `handleActivatePortal`; 409/422 are non-fatal (toast + refetch preview), success invalidates the preview + events queries. **NO WhatsApp button.** **Unchanged:** the 14A pure service, portal lifecycle routes, questionnaires/submission/routing/dashboard/public routes; Twilio/WhatsApp send path entirely untouched.

## Gotchas

Genuinely surprising things that will bite a future contributor.

*   **WhatsApp webhook is fail-closed** — absence of both `WHATSAPP_APP_SECRET` and `TWILIO_AUTH_TOKEN` returns 503 on every call.
*   **Lead Honeypot Field:** The public form must NEVER render an input named `website`; the API silently rejects any submission whose body includes a non-empty `website` (returns synthetic 201 with id `00000000-…` and reference `EMA-PENDING-OK`). New form fields must avoid that name.
*   **Document Deletion Scope:** `DELETE /api/documents/:id` requires `leadId` scope and rejects requests without it.
*   **Step 7 Schema Optionality:** `wantsToUploadDocuments` is `.optional()` in the assessment zod schema even though the UI requires it, so step 6's `type=submit` Continue button isn't silently blocked by zod-resolver before the user has seen the gate. Submit/Finalize buttons enforce the choice via `disabled={!wantsDocs}`.
*   **DocumentUploader allow-list is server-side only:** Frontend MIME/extension allow-list removed (`accept="*/*"`) with a soft hint; server's allow-list in `routes/documents.ts` is the source of truth.
*   **Autofill on WhatsApp input:** Chromium occasionally autofills the WhatsApp field; handled by clearing during tests.
*   **Campaign send returns 202, not 200** (post-6D-3A). UIs that read `tally` from the response need to switch to polling campaign-detail for live counter updates.
*   **Campaign send queue-readiness:** First `/send` call within ~1s of API boot can return 503 (queue not ready). Operator just retries. The scheduler's tick has the same gate, so a scheduled campaign whose fire time lands in the boot-warmup window is silently deferred to the next 30s tick (no audit row written).
*   **Pause is best-effort, not instant** (post-6D-3B). Recipients already claimed by the worker at pause time will still dispatch — only NEW worker claims see the paused state and revert `sending → queued`. Resume re-enqueues every still-`queued` recipient. Pausing an actively-draining campaign may still send up to ~`batchSize=8` more emails before the pause "lands."
*   **Scheduled campaigns remain editable** (post-6D-3B). PATCH accepts both `draft` and `scheduled` status. The dispatch path re-evaluates the audience query at fire time, so last-minute filter changes are honoured. If audience becomes empty between schedule and fire, scheduler reverts to `draft` (audit row written, no operator notification).
*   **Stuck `sending` recipients:** A hard process kill between the atomic `queued→sending` claim and the worker's terminal settle leaves the recipient row in `sending` indefinitely (pg-boss retry no-ops because the claim requires `status='queued'`). Acceptable single-replica MVP gap; manual SQL flip to `failed` if needed.
*   **Email body is stored as raw HTML** (post-6D-2). Sanitiser runs on write AND render — never bypass it. Plain-text WA bodies remain plain text.
*   **Source attribution endpoint is bespoke** (post-6E). `/api/stats/source-attribution` is intentionally outside OpenAPI — frontend fetches via plain `useQuery`, no orval hook. Don't add it to the spec until the shape stabilises.
*   **Legal Modals:** All legal modals currently display placeholder copy awaiting final review. (V1 limitation.)
*   **Orphaned Blobs:** Object storage blobs are not deleted when documents are removed. (V1 limitation.)
