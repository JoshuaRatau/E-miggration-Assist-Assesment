2. Background — what this funnel is

This repository is the lead-capture funnel, not the main EMA platform. It captures prospective clients through a guided assessment and manages them in an admin CRM. The new referral capability lets qualified, consenting leads be passed to a partner law/immigration firm, who then picks them up inside the separate main EMA system. We built only the sending side of that hand-off; the receiving side lives in the other project.

4. Part B — Referral tunnel (what it does)

The tunnel adds a consent-based referral pathway with privacy and safety built in from the ground up.

How it flows

  • Consent: a lead explicitly opts in (POPIA-aligned) to be shared with a partner firm.
  • Match & offer: the funnel matches a suitable firm and offers them a redacted summary (matter type, urgency, region — no name, email, or phone).
  • Accept & open: the firm accepts and is redirected into the main EMA platform via a short-lived, signed link; the applicant's full details are delivered securely to EMA at that moment.
  • Status callbacks: EMA reports progress back (account linked, converted, failed), which the funnel records.
  • Admin board: a new /admin/referrals view lets the team see referral status.

Safety guarantees engineered in

  • Personal data is structurally protected. The referral records table has no name/email/phone columns at all — personal details only ever travel inside a single encrypted, signed hand-off to EMA.
  • Every message is cryptographically signed (HMAC-SHA256) with a shared secret, so neither side can be spoofed.
  • Fails safely. If the shared secret is missing, the tunnel refuses to operate rather than leak anything (returns a "not configured" response).
  • No duplicates, no data loss. Consent creation locks the lead so two clicks can't create two referrals; a "converted" referral can never be accidentally regressed to an earlier state.

6. Open item — action on the main EMA platform (not the funnel)

Action needed  During cross-system testing, the main EMA platform's signed status callback was rejected by the funnel with a signature error. We proved conclusively this is not a funnel problem: a callback signed with the funnel's own key passes the live funnel's signature check, so the funnel's development and production keys match each other and the contract is correct.

Root cause: EMA's copy of the shared secret (REFERRAL_TUNNEL_SECRET) is not byte-identical to the funnel's — a copy-paste drift (typically a stray space or newline). Fix (on the EMA side): re-paste the exact same secret value into EMA's workspace and production secrets, then republish EMA. No funnel change required.
